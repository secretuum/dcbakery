## Безопасность

### A5 — Защита /api/admin/**

**Статус: ✅ ИСПРАВЛЕНО** (коммит `b8bf702`, 2026-07-XX)

- Файл: `middleware.ts` в корне проекта ✅
- Функция `export async function middleware(request: NextRequest)` ✅
- Matcher:
  ```typescript
  export const config = {
    matcher: ["/admin/:path*", "/api/admin/:path*"],
  };
  ```
- Логика:
  1. Пропускает `/admin/login`, `/api/admin/login`, `/api/admin/logout` без проверки
  2. Для non-safe методов (POST/PATCH/DELETE) проверяет `Origin` заголовок против `host` — CSRF защита
  3. Проверяет `access_token` через Supabase `/auth/v1/user` + `isAdminIdentity()`
  4. Если токен истёк — автоматически обновляет через `refresh_token`
  5. При неудаче: API → 401, страница → redirect `/admin/login`
- `/api/admin/**` **реально защищён** ✅

### C13 — PostgREST инъекция в admin.ts

**Статус: ✅ ИСПРАВЛЕНО на уровне API** (коммиты `e7f82cd`, `3957b18`)

#### Проблемный код в admin.ts (строки 639–666):

```typescript
// src/lib/supabase/admin.ts
export async function fetchClientOrderSummaries({
  email,
  phone,
}: {
  email?: string;
  phone?: string;
}) {
  const filters: string[] = [];

  if (email) {
    filters.push(`customer_email.eq.${email}`);   // строка 654 — прямая интерполяция
  }
  if (phone) {
    filters.push(`customer_phone.eq.${phone}`);   // строка 658 — прямая интерполяция
  }

  if (filters.length > 1) {
    params.set("or", `(${filters.join(",")})`);   // результат: ?or=(email.eq.X,phone.eq.Y)
  } else if (email) {
    params.set("customer_email", `eq.${email}`);  // строка 664 — безопасно (отдельный param)
  } else if (phone) {
    params.set("customer_phone", `eq.${phone}`);  // строка 666 — безопасно
  }
}
```

Сама функция `admin.ts` НЕ валидирует email/phone — уязвимость была бы возможна при подаче `email = "foo@bar.com,customer_phone.ilike.*"`.

#### Исправление (в вызывающем роуте):

```typescript
// app/api/profile/orders/route.ts, строки 10–11
const EMAIL_RE = /^[^,()[\]\s]+@[^,()[\]\s]+\.[^,()[\]\s]+$/;
const PHONE_RE = /^\+?\d{10,15}$/;

// строки 38–45: валидация ДО вызова fetchClientOrderSummaries
if (session.email && !EMAIL_RE.test(session.email)) {
  return NextResponse.json({ error: "Bad Request" }, { status: 400 });
}
if (session.phone && !PHONE_RE.test(session.phone)) {
  return NextResponse.json({ error: "Bad Request" }, { status: 400 });
}
```

Регекс `EMAIL_RE` исключает символы `,()[]` — основные векторы PostgREST инъекции.

#### Остаточный риск

`fetchClientOrderSummaries` в `admin.ts` по-прежнему не имеет внутренней валидации. Если функция когда-либо будет вызвана из другого места (например, WhatsApp-бот, admin API) без аналогичной проверки — инъекция возможна снова. Рекомендуется добавить валидацию внутрь функции.
