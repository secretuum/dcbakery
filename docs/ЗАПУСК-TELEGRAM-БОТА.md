## Как это работает (коротко)

```
Заявка с сайта
   ↓
Карточка с кнопками падает в общий Telegram-чат
   ↓
Менеджер жмёт «Подтвердить»  →  статус «Ждёт оплаты», срок оплаты = подтверждение + 4 дня
   ↓                             клиент видит счёт и накладную у себя в кабинете
Бухгалтер в срок жмёт «Оплачено»
   ↓
Менеджер: «В работу» → «Доставляется» → «Выполнен»

Если за 4 дня не оплатили → ежедневный крон ставит статус «Просрочен» и пишет в чат.
```

Кто что может (проверяется в момент нажатия кнопки):

| Действие                     | Админ | Менеджер | Бухгалтер |
|------------------------------|:-----:|:--------:|:---------:|
| Подтвердить / Отклонить      |  ✅   |    ✅    |    —      |
| Отменить                     |  ✅   |    ✅    |    —      |
| Оплачено / Снять оплату      |  ✅   |    —     |    ✅     |
| В работу / Доставляется / Выполнен | ✅ | ✅   |    —      |

Руководитель цеха работает в **веб-админке** (`/admin`), не в боте. Клиенту бот
ничего не пишет — клиент всё видит в личном кабинете на сайте.

Это обычный веб-редактор внутри Supabase — никакой командной строки, ничего

### A2. Разрешить новый статус `overdue` (мини-миграция CHECK)

**Без этого крон просрочки будет падать с ошибкой 500** — БД не даёт поставить
статус, которого нет в списке разрешённых.

**Ничего искать и подставлять не нужно.** Скопируй блок ниже целиком и нажми
**Run**. Он сам найдёт старое правило на колонке `status`, удалит его и создаст
новое — уже вместе с `overdue` (и со всеми старыми статусами, чтобы ни один
существующий заказ не сломался):

```sql
DO $$
DECLARE
  c record;
  status_col smallint;
BEGIN
  -- порядковый номер колонки status в таблице orders
  SELECT attnum INTO status_col
  FROM pg_attribute
  WHERE attrelid = 'orders'::regclass AND attname = 'status';

  -- удаляем любое старое CHECK-правило, стоящее именно на колонке status
  -- (как бы оно ни называлось)
  FOR c IN
    SELECT conname FROM pg_constraint
    WHERE conrelid = 'orders'::regclass
      AND contype = 'c'
      AND conkey = ARRAY[status_col]
  LOOP
    EXECUTE format('ALTER TABLE orders DROP CONSTRAINT %I', c.conname);
  END LOOP;

  -- создаём новое правило со ВСЕМИ статусами
  ALTER TABLE orders ADD CONSTRAINT orders_status_check CHECK (
    status IN (
      'pending_manager_confirmation',
      'change_proposed',
      'confirmed_waiting_payment',
      'overdue',
      'paid',
      'in_progress',
      'delivering',
      'completed',
      'canceled',
      -- старые статусы (могут быть у ранних заказов) — не удалять:
      'new', 'confirmed', 'ready', 'delivered', 'cancelled'
    )
  );
END $$;
```

Если хочешь убедиться, что сработало (необязательно) — выполни отдельно:

```sql
SELECT pg_get_constraintdef(oid)
FROM pg_constraint
WHERE conrelid = 'orders'::regclass AND conname = 'orders_status_check';
```

В ответе будет длинная строка со списком статусов — в ней должно быть `'overdue'`.

## ШАГ C. Крон просрочки на Render

Раз в сутки нужно дёргать endpoint, который помечает неоплаченные вовремя заявки
как «Просрочен» и пишет сводку в чат.

Render → **New → Cron Job** (или в существующем сервисе раздел Cron):

- **Schedule:** `0 6 * * *` (каждый день в 06:00; поставь удобное время)
- **Command:**

```bash
curl -fsS -H "Authorization: Bearer $CRON_SECRET" \
  https://ТВОЙ-ДОМЕН/api/cron/overdue
```

Замени `ТВОЙ-ДОМЕН` на адрес сайта. `CRON_SECRET` должен совпадать с тем, что в env.

Проверить вручную можно из браузера/терминала:
`https://ТВОЙ-ДОМЕН/api/cron/overdue?secret=ТВОЙ_CRON_SECRET` — ответ `{"ok":true,"marked":N}`.

## Сотрудники и роли — как добавлять

Роли раздаются **списком числовых Telegram-id в переменных Render**. Никаких
`@username` — только числовой id (его нельзя подделать, и он не меняется).

### Как узнать Telegram-id сотрудника

1. Сотрудник открывает бота (или общий чат с ботом) и пишет команду **`/start`**.
2. Бот в ответ присылает его **числовой id** и текущую роль (или «нет доступа»).
3. Сотрудник диктует тебе этот id.

### Как выдать роль

Впиши id в нужную переменную на Render (несколько — через запятую) и **передеплой /
рестарт** сервиса, чтобы новые значения подхватились:

| Роль       | Переменная                | Что может |
|------------|---------------------------|-----------|
| Админ      | `TELEGRAM_ADMIN_IDS`      | всё |
| Менеджер   | `TELEGRAM_MANAGER_IDS`    | вести заказ (подтвердить/отклонить/отменить/в работу/доставляется/выполнен), **кроме оплаты** |
| Бухгалтер  | `TELEGRAM_ACCOUNTANT_IDS` | только **Оплачено / Снять оплату** |

Пример на несколько человек:

```
TELEGRAM_MANAGER_IDS = 111111111,222222222,444444444
TELEGRAM_ACCOUNTANT_IDS = 333333333
```

### Важные нюансы

- **Один id — одна роль.** Если случайно вписать id в несколько списков,
  приоритет такой: админ → менеджер → бухгалтер (сработает первый).
- **Убрать сотрудника** = удалить его id из переменной и передеплоить. Доступ
  пропадает сразу, история его действий остаётся в `bot_audit_log`.
- **Кого нет ни в одном списке** — бот на кнопки не реагирует (для него «нет доступа»).
- **Руководителя цеха в бота добавлять не нужно** — он работает в веб-админке `/admin`.
- После изменения любой переменной обязательно **рестарт/redeploy** — Render
  подхватывает env только при старте.

## ⏸ Отложено (блокер — решается отдельно)

**Накладная по форме З-2.** Сейчас в кабинете отдаётся упрощённая накладная. Чтобы
привести к официальной казахстанской форме З-2, нужно твоё решение по
**«номенклатурному номеру»**: в присланном образце коды (344, 1142, 1145…) — это
коды **1С**, а не iiko (я проверял: в iiko 344 = «Анчоусы», 1142 вообще нет).
Варианты: длинный iiko-код / пусто / коды 1С (нужен источник). Пока не согласуем —
форму не строю. Утилита «сумма прописью» уже готова (`src/lib/number-to-words.ts`).

## Если что-то не так

- **Карточка не приходит в чат** → проверь, что бот в группе и `TELEGRAM_GROUP_CHAT_ID`
  = id этой группы (отрицательное число).
- **Кнопки не реагируют** → id нажавшего нет в списках ролей, либо не сделан рестарт после правки env.
- **Крон отвечает 500** → не выполнен шаг **A2** (статус `overdue` не разрешён).
- **Крон отвечает 401** → `CRON_SECRET` в curl не совпадает с env.
